# 店舗補充管理 要求分析 Step 0

<!-- constrained-by ../../../docs/incremental-modeling.md#stage-0-scope-sketch -->

## 1. 業務背景

店舗運営部では、各店舗の次回補充予定日と運営上の担当組織を継続的に見直している。現状は担当者が表計算ファイルとメールで補充予定を調整しており、補充予定の変更理由や組織変更時の確認経路が追跡しづらい。

本ステップでは、詳細な画面・データ項目・API は決めず、最初に扱う業務領域と BUC の名前だけを合意する。

## 2. 対象スコープ

| 項目 | 内容 |
|---|---|
| 業務領域 | 店舗運営 |
| 初回対象 BUC | 店舗補充情報の維持管理 |
| 主な目的 | 店舗の補充予定と担当組織をレビュー可能な形で管理する |
| 対象外 | 発注実行、在庫引当、配送計画、外部倉庫連携 |

## 3. 業務要求

| ID | 要求 | 優先度 | 備考 |
|---|---|---|---|
| R-001 | 店舗補充管理を店舗運営業務の一部として扱えること | Must | BUC の上位業務を明示する |
| R-002 | 補充予定変更と担当組織変更を今後の分析対象に含められること | Must | usecase は次ステップで確定する |
| R-003 | 画面やデータ設計に入る前に、業務名と境界を関係者がレビューできること | Should | 粗い BUC 図で確認する |

## 4. 前提と制約

- 店舗運営部を最初の業務オーナーとする。
- 対象は社内業務であり、顧客向け UI は含めない。
- この段階ではエンティティ、画面、API、状態遷移を定義しない。

## 5. レビュー観点

- BUC 名が業務担当者に通じる粒度になっているか。
- 対象外にした発注・配送・在庫引当が今回の検討から漏れても問題ないか。
- 次ステップで actor と usecase を洗い出せるだけの業務境界になっているか。

## 6. 次ステップへの確認事項

- 誰が補充予定を変更するのか。
- 担当組織の変更は誰が起票し、誰が承認するのか。
- 業務担当者が「1 つの操作」と認識している単位は何か。
